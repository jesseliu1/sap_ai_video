# 定位只要20分钟，问清楚却花了40分钟：SAP AMS 最隐蔽的时间黑洞

早上九点，你刚打开电脑，工单系统里弹出一条新Incident：

> "保存采购订单的时候跳出来一个红色报错，好像说什么科目要分配，截图发你了，麻烦帮忙看一下，这批料今天要下单，比较急。"

用户是认真的：有操作、有截图，还说明了紧迫程度。

但你打开截图看了20秒：弹窗只截到上半截，报错的消息号正好被挡住，看不清到底是哪个错。凭"科目要分配"这几个字，你心里大概有数——多半是某个成本要素科目没带出成本中心，类似 KI235 那一类——但到底卡在哪一行、哪个科目，光看这半张图定不了。

接下来要干什么，你门儿清：给Key User打电话。

电话拨过去，你一项一项问：是测试机还是正式机？这张单的科目分配类别选的是什么？是只有你这张单报错，还是别人下单也一样？最近这个成本中心、相关科目动过配置没有？

对方那头：系统分不太清，"应该是正式的吧"；科目分配类别没听懂在问什么；别人有没有他没留意；配置是啥他也说不上来。

来回问了几轮，你能确认下来的，其实只有一条：是PRD。

你挂了电话，看了一眼表：9:38。

问题本身，可能20分钟就能定位。但把问题的完整上下文搞清楚，你已经花掉将近40分钟。这种事，几乎每个SAP AMS顾问每周都会经历不止一次。

（上一篇，我把十年工单喂进了一个 RAG，结果撞了墙：历史是座空山，因为没人在关单那一刻把东西真正留下来。撞出来的那句结论是——**喂给 AI 的入口质量，决定了 AI 的上限**。这一篇，就把镜头从"历史的入口"挪到"当下的入口"：一张新单刚进来时，到底差了哪几样东西，才害得后面的人多花一倍时间去补。从频率最高、也最容易在入口出问题的 Incident 处理开始拆。）

## 40分钟到底耗在哪

不是顾问不专业，也不是Key User不认真。

真正的问题是：用户描述问题的思维框架，和顾问分析问题所需的信息框架，天然存在结构性错位。

用户的思维是："我遇到麻烦了，你来帮我解决。"

而顾问要往下分析，需要的是一整套结构化的上下文——这套上下文，也正是一张Incident能不能进入分析阶段的最低标准：

- 问题描述：发生了什么，具体到操作和现象，不是"不能用""有问题"
- 操作路径：事务码、操作步骤、到哪一步出的问题
- 报错信息：完整截图，带系统消息和错误码（不被截断）
- 系统环境：客户端编号、系统（QAS/PRD）、登录用户
- 发生时间：第一次出现的时间，是否持续，是否有规律
- 影响范围：只有提单人，还是整个业务组，还是跨部门
- 业务影响：能否继续操作，是否被卡住，紧急程度
- 最近变更：近期是否有上线、主数据修改、权限调整
- 期望结果：用户希望最终能做到什么

用户提了工单、附了截图，但顾问真正需要的上下文，往往还是缺一半。缺其中任何一项，分析过程里就至少要多一次往返。而用户填不全，是常态，不是用户的错。

## 为什么这个缺口一直补不上

不是团队不知道工单质量重要。很多SAP AMS团队早就有提单规范、有模板、有入口checklist。

问题是：规范有，但执行成本太高，难以持续。

**对用户来说**，"按顾问要的方式描述"是本职工作之外多花的时间。他大致清楚你要哪个系统、哪一步、把报错完整截下来，但他遇到问题时的诉求是"赶紧有人接手"，而不是"替顾问把诊断材料准备好"。问题不在于他不知道，而在于他不一定愿意为别人的排查流程花自己这份时间。

**对顾问来说**，"把追问写到位"同样很费时间。该问什么，顾问一清二楚；难的是把一条追问写得让非技术的业务人员一次就回到点上——既要问全，又要看得懂，还要语气得体、不像在质问。这套语言每张单都得重新组织一遍；写得潦草，用户回不到点上，就得再来一轮。理论上信息不全该先追问再分派，可追问要等回复，等待期间SLA还在走，来回几次之后，顾问有时候干脆带着残缺信息往下走，赌一把——尤其是那种已经稳了一段时间的维护项目：做久了反而越容易松手，总觉得自己早把这个客户摸透了，这一单大概也不会跑偏。

**缺信息带来的返工，往往算在顾问头上。** 分析到一半发现方向错了，或者测试阶段才暴露信息偏差——这些返工很少被追溯到提单入口，更多是顾问默默扛下来。

所以这不是培训用户、或者催顾问"追问更彻底"就能解决的问题。两边其实都大致知道该有哪些信息，真正卡人的是"把它做到位"要付出的时间——用户补材料要时间，顾问写追问也要时间。这份时间，过去只能靠顾问一张单一张单地硬扛。

## AI的第一步：不是补全工单，是把这40分钟的"组织语言"接走

很多人对AI在工单里的想象是：自动读单、自动分析、自动给方案，最好自动修复。方向完全错了。

还有一种想象也不准确：把用户残缺的描述自动"补全"成完整工单。用户没写的信息，AI也变不出来。

AI在Incident入口最有价值的第一步，不是凭空生成信息，而是把顾问和用户之间那段"信息获取过程"做得更快、更精准、更少摩擦——也就是开头那40分钟里，真正被消耗掉的那部分。

具体说，AI做两件事：

- **整理**：把用户已经给的内容，按标准工单格式梳理清楚，让顾问不用自己去啃原始描述。
- **追问**：对照完整Incident标准，识别哪些字段还缺、缺的为什么影响分析，然后起草一条追问消息，顾问确认后发给用户。

注意它省的是哪一截：顾问反复组织追问语言、用户理解"到底在问什么"的那段时间。它把过去"边想边问的多轮电话"，换成"一次问全的书面追问"。说实在的，用户配合不配合、看不看得懂，仍然是变量，这条不保证一轮就问到位——但至少顾问不必每张单都从头斟酌该问什么、怎么问，用户也不必在电话里被零散地追着问。

这是改造最轻、能马上铺开的一层：不接客户SAP环境，不改ITSM主流程，只就着用户已经给的材料，做结构化和缺口识别。但你现在看到的，还只是把它当个通用工具在用——照一份谁都拿得到的清单干活。真正能在团队之间拉开差距的，是有没有人能把它训成一版真正贴合你这个项目的智能体——同样的起点，训出来合不合用、懂不懂你这个客户，差得远。这一层再往里走，价值更高，要过的边界也更多，不是谁都做得了，那是后话。

（这里得插一句边界：把用户的原始材料直接丢给AI，数据能不能用、合规这道线怎么划，本身是前提——这一篇先不展开，后面会有一篇专门正面谈"客户不让工单进 AI，到底该怎么把'不准'谈成'可控试点'"。）

**AI的输入**，就是用户为描述这个问题留下的所有痕迹：工单原始描述、邮件、企业微信／Teams聊天截图、报错截图OCR、电话录音转写、L1的简短备注。不用整理、不用分类，直接丢给它。

**AI的输出**，是四样东西：

1. 现有信息的结构化摘要——把用户提供的内容理成标准工单格式，L1扫一眼就知道什么模块、什么现象、已知的系统和操作路径；
2. 缺失字段清单——带"缺这个会让分析卡在哪一步"的说明，而不是干巴巴一句"信息不全，请补充"；
3. 一条可直接发给Key User的追问草稿——问题有针对性、语气清晰，顾问确认后直接发出；
4. 仅供参考的初步模块分类——给出可能涉及的模块和问题类型，不作为最终分类依据。

## 回到开头那张工单

抽象标准落到开头"保存采购订单报错"这张单上，AI实际吐出来的大概长这样：

**① 结构化摘要**
- 模块：MM（采购订单）
- 现象：保存采购订单时弹红色报错，提示"科目需要分配"，操作中断
- 系统：PRD（来自后续电话补充）
- 报错信息：截图被截断，消息号不可见；据描述疑似成本要素科目缺成本对象分配（KI235一类），待确认
- 操作路径：保存采购订单时报错（事务码、出错行项目未提供）
- 紧急程度：提单人标注"今天要下单"；据现有信息初步评定优先级为「中等」，待确认影响范围后定

**② 缺失字段清单**
- 报错信息不完整：截图被截断，看不到消息号和完整文本 → 无法确认是不是KI235、卡在哪个科目
- 操作路径不完整：缺事务码、缺出错的具体行项目和科目 → 无法复现、无法定位
- 影响范围未知：只确认提单人，未确认别人下单是否也报错 → 无法判断个案还是配置面问题
- 最近变更未知：未说明相关成本中心/科目近期是否动过配置 → 无法排除变更引起

**③ 追问草稿——AI已经按邮件格式拟好，扫一眼就能发**

> **主题：【采购订单保存报错｜今日下单】请补充信息以确认优先级、加快定位**
>
> 您好，
>
> 收到您反馈的采购订单保存报错，已经在跟进。依据目前提供的信息，先把本单紧急程度初步评定为「中等」；为进一步确认优先级、也为了尽快帮您定位，还得麻烦您补几项信息：
>
> **1. 完整的报错截图（这步最影响定位）**：报错出现时先别急着点掉——
> 　· 如果红字显示在屏幕最下方的状态栏，用鼠标**双击**那行红字，会弹出完整说明，再截图；
> 　· 如果是弹窗，把整个弹窗连标题一起截下来；
> 　· 截图时把整个SAP窗口都框进去（顶上的事务码栏、底下的状态栏都带上），别只截中间一块；开头那个"字母+数字"的消息号（类似 KI235）一定要能看清。
> **2. 操作场景**：您是在创建还是修改采购订单？报错时停在哪一行、对应哪个科目？
> **3. 影响范围**：这个报错只有您这张单遇到，还是别人下同类单也一样？
> **4. 最近变更**：这批料对应的成本中心／科目，最近有没有调过配置？
>
> 这几项一到，我们就能确认优先级、马上往下排查。麻烦您了，谢谢！
>
> ——XXX｜SAP运维支持

**④ 初步分类参考（仅供参考）**
可能涉及：成本要素科目缺成本对象（成本中心/内部订单）分配，或科目分配类别设置问题，需消息号和具体科目确认。

顾问要做的，从"自己把这一整套组织出来"，变成"扫一眼、改一两处、点发送"。你自己从头写这么一封——要问全、要让业务看得懂、语气还得拿捏——怎么也得10到15分钟；AI出这版草稿是几秒钟的事，你过一眼、改一两处就能发。

不妨拿上面这条追问草稿，对照你平时怎么问同类问题——差距大多不在"你不会问"，而在"每张单都得把它从头再写一遍"。

## 它能干到哪，到哪为止

这一步省下来的，是顾问理摘要、写追问的时间。过去这截只有两条路：自己干，盯着一堆原始描述一项项理、再字斟句酌写追问，每张单重来一遍；或者派给初级顾问，可光是跟他讲清楚"这张单缺什么、该怎么问"，有时比自己写还慢，写完还得回头改。

现在这部分时间几乎都省了。AI把摘要理好、追问写好，只到"可以发"为止——给的是草稿，不是定论。可这一眼复核，不是它没干完留下的尾巴：这草稿能不能发、发出去对不对，最后都压在你那一眼上，而这一眼，恰恰是你专业判断真正上场、真正值钱的地方。

与其说它"接近一个有经验的顾问"，不如说它是一个永不偷懒、严格照清单走的稳定L1：哪个字段缺了、缺这个会让分析卡在哪一步、该用什么措辞一次把问题问清楚——这些它做得又快又不走神。

| AI 能做 | 顾问需要做 |
|---|---|
| 把已有信息理成结构化摘要 | 复核摘要，确认没理偏、没漏项 |
| 识别缺失字段，说明缺失为什么影响分析 | 判断根因方向，决定这单怎么往下查 |
| 起草追问消息，提高获取信息的效率 | 过一眼追问，确认无误后发给Key User |
| 给出初步模块分类参考 | 核定最终分类与优先级 |
| 检索、整理相似的历史工单 | 跟业务确认影响范围，拍板关单、对结果负责 |

**谁来审？** AI生成的摘要、缺失清单、追问草稿，都必须经L1或对应模块顾问确认，才能进正式流程或发给用户。

把识别缺口、从头组织追问这层最费神的活替你打好；但拍板这条线，它永远不越。

## 对乙方团队，这件事到底值在哪

如果只是"解救一个顾问的早晨"，它其实不值得交付负责人认真对待。它真正的价值，是把"入口质量"从一件靠人临场发挥的事，变成一件可以被安装、被复制、被审查的能力。

它减少的是：

- 顾问反复组织追问、用户没回到点上又得再问一轮的往返（开头那40分钟的主体）
- L1分派时因描述不清导致的误分类
- SLA被入口沟通摩擦白白消耗的时间

它让这几件事更容易长期坚持：

- 工单入口的信息完整性，不再完全依赖某个顾问当时的状态和临场组织能力
- 每张工单都有一份格式统一的摘要，AMS Manager 做工单 review 的成本随之降低
- 后续 RCA 和知识沉淀，因为输入更完整，质量更高

一张工单的入口质量，影响的不只是这一张票，而是后面几乎所有环节——分类、分派、SLA、分析、测试、关闭、复盘、知识沉淀。**入口多补一格，下游就少返一次工。**

## 下一篇：关单那一刻，怎么让这张单对三年后的人还有用

把入口补全，这张单处理得更顺——但这还只解决了"当下这张票"。真正决定这些工单三年后还有没有价值的，是**关单那一刻**：你往"解决方案"那一栏留下的，是下一个人能照着做的东西，还是一句"已处理"的正确的废话。

这正是开篇那座空山的根——知识从没真正回流进系统。下一篇，就讲怎么在关单那一刻把它补上：这是 L1 真正的核心建设动作，也是后面 RCA、月报、知识沉淀全都要踩的地基。

（至于"数据能不能用、谁来审、出了错谁负最终责任"这些更硬的治理与边界问题，后面会有一篇专门正面谈：客户不让工单进 AI，哪些是真红线、哪些只是惯性，怎么把"不准"谈成"可控试点"。）
