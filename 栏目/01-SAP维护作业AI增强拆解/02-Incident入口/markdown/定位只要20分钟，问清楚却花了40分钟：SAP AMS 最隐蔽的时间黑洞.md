# 定位只要20分钟，问清楚却花了40分钟：SAP AMS 最隐蔽的时间黑洞

真正消耗顾问时间的，很多时候不是定位问题，而是把问题问清楚。

你早上打开工单系统，看到一条新 incident：

> 保存采购订单的时候跳出红色报错，好像说什么科目分配。截图发你了，麻烦帮忙看一下，这批料今天要下单，比较急。

用户不是敷衍。

他已经说了业务动作、上传了截图、还说明了紧急程度。

但你打开截图看了20秒，发现弹窗只截到一半，消息号被遮住，采购组织、公司代码、物料号、科目分配类别、是否服务类采购，统统没写。

你不能直接分析，只能先追问。

于是你发第一轮：

> 请补一下采购订单号、公司代码、完整报错截图。

过了十分钟，用户补了截图，但采购订单还没创建成功，没有 PO 号。

你再问：

> 那请提供创建时的供应商、采购组织、物料号、数量、科目分配类别，以及是否有成本中心或内部订单。

又过十五分钟，Key User 转了一段业务群聊天给你。

里面有一句“是不是上次那个料的问题”，还有两张模糊截图。

到这里，真正的定位可能只要 20 分钟。

但在这之前，问清楚这张单，已经花掉 40 分钟。

这就是 SAP AMS 里最隐蔽的时间黑洞：

**问题不是不会处理，而是入口不完整。**

## 乙方其实控制不了客户怎么提单

这里要先把话说准。

乙方通常控制不了客户原始提单入口。

最终用户怎么描述、Key User 怎么转述、客户 ITSM 字段怎么设计、邮件里写不写复现步骤，这些很多时候都在客户侧。乙方不能指望客户从今天开始每张 incident 都写得像顾问分析记录一样完整。

所以我们说“入口补全”，不是说乙方要去改造客户原始提单入口。

更现实的做法是：

> 在乙方接单后的受理入口，建立一个低成本的完整性检查层。

也就是在这张单进入顾问正式分析之前，先把它整理成一个顾问能看、L1 能分、AMS Manager 能回看、未来知识库能沉淀的结构。

客户原始输入可以不完美。

但脏输入不能直接流进分析、RCA、月报和知识沉淀。

这就是 AI 在 incident 入口最值得先做的一刀。

## 一张 incident 到底缺什么

我们先看这张单。

原始输入是：

> 保存采购订单时报错，好像说科目分配，截图发你了，今天要下单，比较急。

这句话里已经有一些信息：

- 业务动作：保存采购订单
- 现象：红色报错
- 大概方向：科目分配
- 紧急程度：今天要下单
- 附件：截图

但对顾问来说，还缺很多关键字段：

| 缺失项 | 为什么重要 |
|---|---|
| 事务码 / 操作路径 | 是 ME21N 创建 PO，还是其他采购相关动作 |
| 公司代码 / 采购组织 / 工厂 | 决定配置、组织层级和影响范围 |
| 物料号 / 物料组 | 判断是否与物料主数据、科目确定、采购类型有关 |
| 科目分配类别 | 判断成本中心、内部订单、资产、项目等成本对象 |
| 完整报错文本 / 消息号 | 没有消息号，无法准确定位标准校验或增强逻辑 |
| 是否新问题 / 是否多人受影响 | 决定优先级和是否存在批量影响 |
| 是否可复现 | 决定是偶发数据问题还是稳定流程问题 |

这些字段不是为了“填表好看”。

它们决定了后面怎么分派、怎么判断优先级、怎么查配置、怎么判断影响范围。

如果这些信息不全，L2 顾问一上来就被迫做两件低价值的事：猜问题、追材料。

## AI 在这里不是自动解单，而是做入口完整性检查

这一步，AI 不需要接 SAP，也不需要写回工单系统。

它只需要拿到用户原始描述、截图文字说明、Key User 转述、邮件或 IM 片段，然后输出三类东西：

1. **结构化摘要**：这张单目前能确定什么；
2. **缺失字段清单**：还缺什么，为什么缺了会影响分析；
3. **追问草稿**：下一轮应该一次性向用户或 Key User 问清楚什么。

用上面的例子，AI 可以整理出这样的受理摘要：

| 字段 | 当前信息 | 状态 |
|---|---|---|
| 作业类型 | 采购订单保存报错 | 已知 |
| 疑似模块 | MM-PUR / FI-CO 相关 | 初步判断 |
| 报错方向 | 科目分配相关 | 不完整 |
| 业务紧急度 | 今日需下单 | 已知 |
| 公司代码 / 工厂 | 未提供 | 缺失 |
| 物料 / 物料组 | 未提供 | 缺失 |
| 科目分配类别 | 未提供 | 缺失 |
| 完整报错 / 消息号 | 截图不完整 | 缺失 |
| 影响范围 | 未知是否多人受影响 | 缺失 |

然后给出一段追问草稿：

> 为了判断该采购订单保存报错的原因，请协助补充以下信息：
>
> 1. 创建采购订单时使用的公司代码、采购组织、工厂；
> 2. 供应商、物料号或物料组；
> 3. 是否使用科目分配类别，如 K（成本中心）、F（订单）、A（资产）等；
> 4. 完整报错截图，尽量包含消息号和完整文本；
> 5. 是否只有这一笔采购出现，还是同类物料 / 同公司代码下多人都遇到；
> 6. 如果是紧急采购，请说明最晚需要完成下单的时间。
>
> 当前看，问题可能与采购订单科目分配或成本对象校验有关，但需要以上信息确认后才能判断根因。

这段东西并不神奇。

但它稳定、完整、不会漏问。

更重要的是，它把顾问从“每张单都要从头组织追问”里解放出来。

## 顾问价值没有消失，而是后移了

这一步很容易被误解成“AI 在替顾问处理 incident”。

不是。

AI 做的是入口整理，不是根因判断。

它能帮你把一张含糊的单整理成可分析输入，但它不能决定：

- 到底是配置问题、主数据问题、权限问题还是增强问题；
- 是否需要查 OBYC、科目确定、成本中心有效期、采购类型配置；
- 是否涉及 FI / CO / MM 跨模块；
- 是否需要 workaround；
- 是否需要走 change；
- 是否可以关闭。

这些都还是顾问判断。

只是顾问不再把时间浪费在“这张单到底在说什么”上。

顾问真正值钱的地方，是从结构化输入之后开始：判断方向、确认影响范围、找根因、给方案、推动业务确认。

这也是我更愿意把这一步叫作：

> 乙方受理入口的 AI 完整性检查层。

它不是自动解单。

它是让顾问更快进入真正需要顾问的地方。

## 对 AMS Manager 来说，这不是小工具，是管理动作

如果只看单张 incident，这件事像是给顾问省几分钟。

但站在 AMS Manager 视角，它更像一个管理动作。

因为入口补全一旦标准化，会带来几个连锁效果：

- L1 分派更准，少把 MM 单丢给 FI，再绕回来；
- L2 顾问少做重复追问，响应时间更稳定；
- SLA 计时里的无效等待更容易被识别；
- RCA 不再从一堆碎片材料里倒推；
- 月报里“用户描述不清导致延迟”的问题能被量化；
- 知识沉淀不再只留下“已处理”。

这也是为什么我说，第一层不等于低价值。

它虽然不接系统、不改配置、不做深度集成，但它服务的是后面更高价值的判断：根因分析、影响范围、变更风险、服务治理。

入口补全不是 L1 的杂活。

它是 L2/L3 高价值判断之前的输入质量控制。

## 一个最小检查清单

如果一个 SAP AMS 团队想先试这件事，不需要一上来做系统集成。

先拿最近 20 张 incident 做一个最小验证就够了。

每张单让 AI 输出下面 6 项：

1. 当前问题摘要；
2. 疑似模块 / 作业类型；
3. 已知事实；
4. 缺失字段；
5. 缺失字段为什么影响分析；
6. 给 Key User 的一次性追问草稿。

然后让顾问只评三件事：

- 摘要有没有理偏；
- 缺失字段有没有漏；
- 追问草稿能不能直接发。

如果 20 张里，有一半以上能减少一轮追问，这个场景就值得继续做。

如果 AI 总是问不到点上，说明不是模型没用，而是你们还需要先补一层本团队的字段规范和分类规则。

这就是低风险试点。

不碰客户生产系统，不自动关单，不替顾问判断根因，只验证一件事：

> AI 能不能把 incident 受理入口的脏输入，整理成顾问愿意接着分析的材料。

## 最后

我以前做历史工单 RAG 时，最深的感受是：很多工单之所以三年后没有价值，不是因为系统里没存，而是因为当时入口和过程就没留下可复用的信息。

所以 SAP 维护团队做 AI，第一步不是急着问“能不能自动解单”。

更该问的是：

> 从今天开始，每一张新进来的单，能不能先变成一份未来还能被人和 AI 读懂的记录？

如果这个问题不解决，后面的 RCA、知识库、月报、服务治理，都只是在脏输入上继续加工。

入口脏，后面所有 AI 动作的上限都会被限制。

所以在 SAP 乙方交付里，AI 最值得先切的一刀，是乙方受理入口。
