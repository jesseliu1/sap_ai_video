# Incident处理里，AI第一步不是自动解单，而是先把工单入口变完整

上一篇我们讲了：AI在SAP乙方团队里真正的位置，不是推翻现有流程，而是在已有流程上降低执行成本。这一篇，我们从第一类作业开始拆：Incident处理。这是SAP AMS里出现频率最高、但也最容易在入口就出问题的一类作业。

## 一张还不够用的工单

早上九点，你刚打开电脑，工单系统里弹出一条新Incident：

> “保存采购订单的时候跳出来一个红色报错，好像说什么科目要分配，截图发你了，麻烦帮忙看一下，这批料今天要下单，比较急。”

用户是认真的。有操作，有截图，还说明了紧迫程度。

但你打开截图，看了20秒：弹窗只截到上半截，报错的消息号正好被挡住了，看不清到底是哪个错。凭“科目要分配”这几个字，你心里大概有数——多半是某个成本要素科目没带出成本中心，类似 KI235 那一类——但到底卡在哪一行、哪个科目，光看这半张图定不了。

你知道接下来要干什么：给Key User打电话。

电话拨过去，你一项一项问：是测试机还是正式机？这张单的科目分配类别选的是什么？是只有你这张单报错，还是别人下单也一样？最近这个成本中心、相关科目有没有动过配置？

对方那头：系统分不太清，“应该是正式的吧”；科目分配类别没听懂是在问什么；别人有没有他没留意；配置是啥他也说不上来。

来回问了几轮，你能确认下来的，其实只有一条：是PRD。

你挂了电话，看了一眼表：9:38。

问题本身，可能20分钟就能定位。但把问题的完整上下文搞清楚，你已经花掉了将近40分钟。

这种事，几乎每个SAP AMS顾问每周都会经历不止一次。

## 问题出在哪里？

不是顾问不专业，也不是Key User不认真。

真正的问题是：用户描述问题的思维框架，和顾问分析问题所需的信息框架，天然存在结构性错位。

用户的思维是：

> “我遇到麻烦了，你来帮我解决。”

而顾问分析问题，需要的是：

- 操作路径（哪个事务码、哪个步骤）
- 报错信息（完整截图，带上系统消息和错误码）
- 系统环境（客户端编号、系统编号、登录用户）
- 发生时间（方便关联日志和批处理记录）
- 影响范围（只有提单人，还是整个业务组）
- 业务影响（能不能继续操作，还是完全卡住）
- 最近变更（近期是否有上线、主数据修改、权限调整）
- 期望结果（用户希望达到什么目标）

用户提了工单，也附了截图，但顾问真正需要的上下文，往往还是缺一半。

这个gap本身不难理解。难的是：它每天都在以“一次次电话、一封封追问”的形式，悄悄消耗着顾问的时间——而且很难根除。

## 为什么这个问题一直没被解决？

不是团队不知道工单质量重要。

很多SAP AMS团队早就有提单规范、有工单模板、有入口checklist。

问题是：规范是有的，但执行成本太高，难以持续。

**一、对用户来说，“按顾问要的方式描述”是一笔额外的时间成本。** 很多Key User其实大致清楚你想要什么——哪个系统、哪一步、把报错完整截下来。但把这些一项项补齐、说明白，是在他自己本职工作之外，多花的时间。他遇到问题时的诉求是“赶紧有人接手”，而不是“替顾问把诊断材料准备好”。所以问题不在于他不知道，而在于他不一定愿意为别人的排查流程，花自己的这份时间。

**二、对顾问来说，“把追问写到位”同样很费时间。** 该问什么，顾问门儿清。难的是把一条追问写得让Key User一次就回到点上——既要问全，又要让非技术的业务人员看得懂，还要语气得体、不像在质问。这套语言，每张工单都得重新组织一遍；写得潦草，用户回不到点上，就得再来一轮。理论上信息不全该先追问再分派，但追问要等回复、等待期间SLA还在走，来回几次之后，顾问有时候干脆带着残缺信息往下走，赌一把。

**三、缺信息带来的返工，往往被算在顾问头上。** 分析到一半发现方向错了，或者测试阶段才暴露信息偏差——这些返工成本，很少被追溯到提单入口质量，更多是由顾问默默承担。

所以这不是靠培训用户就能解决的问题，也不是靠催顾问“追问更彻底”能解决的问题。

两边其实都大致知道该有什么信息。真正卡人的，是“把它做到位”要付出的时间——用户把材料补齐要花时间，顾问把追问写清楚也要花时间。这份时间成本，过去只能靠顾问一张单一张单地硬扛，很难低成本地持续下去。

## 一张完整的Incident，需要哪些信息

在讲AI怎么介入之前，先把标准说清楚。

不管有没有AI，一张能进入分析阶段的Incident，至少需要以下信息：

- **问题描述**：发生了什么，具体到操作和现象，不是“不能用”或“有问题”
- **操作路径**：事务码、操作步骤、到哪一步出了问题
- **报错信息**：完整截图，包含系统消息和错误码（不被截断）
- **系统环境**：客户端编号、系统（QAS/PRD）、登录用户
- **发生时间**：第一次出现的时间，是否持续，是否有规律
- **影响范围**：只有提单人，还是整个业务组，还是跨部门
- **业务影响**：能否继续操作，业务是否被卡住，紧急程度
- **最近变更**：是否有近期上线、主数据修改、权限调整
- **期望结果**：用户希望最终能做到什么

用户填不完整这件事本来就是常态，不是用户的错。

这张清单的用处在另一个方向：它是顾问判断“这张工单能不能开始分析”的最低标准，也是后面讲AI介入时的对照基准。缺少其中任何一项，都意味着分析过程中会多至少一次额外往返。

## AI第一层介入：提高追问效率，而不是填补空白

很多人对AI在工单场景里的想象是：AI自动读取工单，自动分析问题，自动给出解决方案，最好自动修复。

这个方向完全走错了。

还有一种想象同样不够准确：AI把用户写的残缺描述，自动“补全”成一份完整工单。

用户没写的信息，AI也变不出来。

现阶段AI在Incident处理里最有价值的第一步，不是“补全工单”，而是：把顾问和用户之间的信息获取过程，变得更快、更精准、更少摩擦。

具体说，AI做两件事：

**整理：** 把用户已经提供的内容按标准工单格式梳理清楚，让顾问不用自己去啃乱七八糟的原始描述。

**追问：** 对照完整Incident的信息标准，精确识别哪些字段还没有、缺的东西为什么影响分析，然后起草一条追问消息；顾问确认后直接发给用户，用户一看就知道该回什么，不需要再被打一次电话。

这个过程节省的不是“凭空生成信息”的时间，而是顾问组织追问语言、用户理解问题是什么、来回沟通摩擦的时间。

这是第一层嵌入深度：不需要接入客户SAP环境，不需要改造ITSM主流程，只需要基于用户已经提供的材料，做信息结构化和缺口识别。

## AI的输入是什么？

用户为了描述这个问题而留下的所有痕迹：

- 工单系统里写的原始描述
- 用户发来的邮件
- 企业微信 / Teams 聊天记录截图
- 报错截图的OCR文字
- 电话沟通后的录音转写
- L1在电话里记的简短备注

不需要整理，不需要提前分类，直接丢给AI。

## AI的输出是什么？

**1. 现有信息的结构化摘要**  
把用户提供的内容整理成标准工单格式，让L1扫一眼就知道：什么模块、什么现象、已知的系统和操作路径是什么。

**2. 缺失字段清单（附说明）**  
对照完整Incident标准，识别哪些字段还没有——不是干巴巴一句“信息不全，请补充”，而是“缺少影响范围（无法判断是否P1/P2）”“缺少最近变更记录（无法排除配置变更引起）”这种带分析依据的说明。

**3. 追问草稿**  
基于缺失字段，起草一条可以直接发给Key User的追问消息，问题有针对性，语气清晰，用户收到知道该回什么。顾问确认内容后直接发出，不需要从头组织语言。

**4. 初步分类参考**  
根据已有信息，给出可能涉及的模块和问题类型建议，仅供参考，不作为最终分类依据。

## 回到开头那张工单

抽象的输出标准，落到开头那条“保存采购订单报错”的工单上，AI实际吐出来的大概长这样：

### ① 结构化摘要

- 模块：MM（采购订单）
- 现象：保存采购订单时弹红色报错，提示“科目需要分配”，操作中断
- 系统：PRD（来自后续电话补充）
- 报错信息：截图被截断，消息号不可见；据描述疑似成本要素科目缺成本对象分配（KI235一类），待确认
- 操作路径：保存采购订单时报错（事务码、出错行项目未提供）
- 紧急程度：提单人标注“今天要下单”

### ② 缺失字段清单

- 报错信息不完整：截图被截断，看不到消息号和完整文本 → 无法确认是不是KI235、卡在哪个科目
- 操作路径不完整：缺事务码、缺出错的具体行项目和科目 → 无法复现、无法定位
- 影响范围未知：只确认提单人，未确认是否别人下单也报错 → 无法判断是个案还是配置面问题
- 最近变更未知：未说明相关成本中心/科目近期是否动过配置 → 无法排除变更引起

### ③ 追问草稿（确认后发给Key User）

您好，为了尽快定位这个采购订单保存报错，麻烦再帮忙提供几点：

1）完整的报错截图——把红色提示框整段截下来，要能看到开头的消息号（类似 KI235 这种）和完整文字，不要被挡住；  
2）您是在哪个界面操作的（创建还是修改采购订单），报错时是哪一行、哪个科目；  
3）这个报错只有您这张单遇到，还是别人下单也一样？  
4）这批料对应的成本中心/科目，最近有没有调整过？

收到这些我们可以马上往下排查，谢谢。

### ④ 初步分类参考（仅供参考）

可能涉及：成本要素科目缺成本对象（成本中心/内部订单）分配，或科目分配类别设置问题，需消息号和具体科目确认。

顾问要做的，从“自己把这一整套组织出来”，变成“扫一眼、改一两处、点发送”。

## 谁审核？

L1顾问，或对应模块顾问。

AI生成的摘要、缺失清单和追问草稿，必须经过人工确认，才能进入正式流程或发给用户。

## 边界在哪里？

| AI可以做 | AI不能做 |
|---|---|
| 整理已有信息，生成结构化摘要 | 对最终分类和优先级拍板 |
| 识别缺失字段，说明缺失原因 | 替顾问判断问题根因 |
| 起草追问消息，提高获取信息的效率 | 凭空补全用户没有提供的信息 |
| 给出初步模块分类参考 | 替业务确认影响范围和业务影响 |
| 整理用户历史描述 | 代替顾问关闭工单 |

如果只记一句话，记这句：

> 你等于多了一个随叫随到、水平稳定的资深帮手——它把“整理材料 + 写追问”这件最磨时间的活接走了。

过去这一步，只有两条路。一条是你自己干：盯着一堆乱七八糟的原始描述一项项理出摘要，再字斟句酌写一封既问得全、又让业务看得懂、语气还得体的追问——每张工单都得重来一遍，时间就这么一点点磨没了。另一条是派给初级顾问：可你光是跟他讲清楚“这张单缺什么、该怎么问”，有时候比自己写还慢，写完还得回头改。

现在，这部分时间被彻彻底底省出来了。AI把摘要理好、把追问写好，你要做的只是扫一眼、改一两处、点发送。它不像新人要带、要教、要反复返工，水平稳定，随叫随到。

它强在哪？识别出哪个字段缺了、缺这个会让分析卡在哪一步、该用什么措辞一次把问题问清楚——这种判断本来要靠几年经验才攒得出来，它确实接近一个有经验的顾问。

但它只把活干到“可以发”为止。所有要拍板的动作——定优先级、判根因、确认业务影响、关单——还是你来。

用资深顾问的判断力，把你最耗时间的那截杂活接走；但拍板这条线，它永远不越。

这就是现阶段AI在Incident入口该待的位置。

## 对SAP乙方团队，这件事真正的价值是什么？

### 减少了什么

- 顾问每天重新组织追问语言的时间
- 因为追问不精准、用户回复不到位、需要再问一次的往返
- 因为信息偏差导致的分析返工
- L1分派时因为描述不清导致的误分类
- SLA被入口沟通摩擦白白消耗的时间

### 让什么事变得更容易坚持

- 工单入口的信息完整性，不再完全依赖顾问临场组织能力
- 每张工单都有清晰摘要，AMS Manager的工单review成本降低
- 后续RCA和知识沉淀，因为输入更完整，质量更高

一张工单的入口质量，影响的不只是这一张票的处理效率，而是后面几乎所有环节——分类、分派、SLA、分析、测试、关闭、复盘、知识沉淀。入口多补一格，下游就少返一次工。

但在继续往更深的场景走之前，有一个更根本的问题得先想清楚。

成熟的SAP乙方团队，在真正决定让AI进入交付流程之前，会先问哪几个问题？数据能不能用？输出谁来审？出错谁来负责？这几个判断如果没想清楚，任何场景的AI试点都容易在第一步就卡住。
