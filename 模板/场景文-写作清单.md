# 场景文·写作清单

> 适用：06–10（RCA / Service Request / 定常 / Change / 月报）等「拆解某类 SAP 维护作业」的场景文。
> 来源：栏目「单篇文章固定检查」10 问 + 母文档 §15.13 证据场。
> （V0.8 已按 §0 重定位校准：价值与可复用交付物在前，边界 / 审核降为调味。）

## 固定回答的 10 问（价值与交付物在前，边界审核降为调味）

1. 这个 SAP 维护作业是什么？
2. 当前流程为什么重、慢、容易漏，或者已有流程为什么仍有提效空间？
3. AI 介入后能带来什么价值？（先讲带来什么，把"不能做什么"往后放）
4. AI 能做出哪些可复用交付物？（结构化交付物 / 检查清单 / 方法论资产——让交付物当主角）
5. AI 可以介入哪个节点？
6. AI 的输入是什么？
7. AI 的输出是什么？
8. 属于哪一层嵌入深度？（从 L1 务实起步是策略不是天花板，要让人看见你看得见通往 L2 / L3 的路；不说"越深越高级"）
9. （调味）谁审核？（顾问 / PMO / AMS Manager / 客户关键用户 / 审批角色）
10. （调味）哪条真红线不能碰？（真红线绝不硬来是可信度的锚，但它是佐料不是主梁，一两句带过）

## 第 11 问（证据场，V0.8 重定位：重心是"做成了什么"）

11. 这篇是否配了至少一个贴近现场、脱敏可复现的最小样例 / 交付物样例 / 检查清单，并呈现**完整价值落差**——脏输入 → AI 处理前的问题 → 结构化交付物 → 审核通过 → 可复用模板 / 检查清单？

若没有，优先补上；确实不适用时，需说明原因。

## 场景文必须带出的管理视角

- 不只是给顾问减负；要让能雇你的老板看见：这个场景如何减少返工、降低重复工时、改善交付留痕、提升团队复用能力。
- 从 L1 切入时，必须说明它服务哪个更高价值动作：L2 顾问少追问、RCA 更稳、Change 更安全、月报更有治理信号。
- 如果是 06 / 08 / 09 / 10 这类依赖入口质量的场景文，固定带一句入口前提，不展开：

> 本文默认工单入口已经过结构化整理；若入口仍是脏的，后续 RCA、月报和知识沉淀都会被限制上限——请先看开篇与《入口篇》。

> 注：07（Service Request）本身就是做补料 / 改善入口，不带此前提句。

## 定稿前自检

- 有没有先讲价值，而不是先讲边界？
- 有没有一个可见交付物 / 样例 / 检查清单？
- 边界有没有降为一两句调味，而不是堆防御密度？
- 读完后，能雇你的人会不会更相信：你不是只懂概念，而是真能把 SAP 维护里的一个具体作业做出东西？
